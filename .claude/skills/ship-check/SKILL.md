# Ship Check

## Мета
Знайти все, що завалить здачу, поки ще є час виправити. Перевіряємо ПРОД, не локалку.

## Чеклист
Пройди по пунктах по черзі. Для кожного попроси в користувача підтвердження
або доказ (скріншот, URL, вивід команди) — не приймай «та все ок» на слово.

1. **Прод-URL живий.** Відкривається в інкогніто-вікні без авторизації
   (або є чітка інструкція для входу).
2. **Ядро наскрізь на проді.** Вставити реальний хаотичний текст → отримати структуровані
   задачі з пріоритетом/часом/дедлайном → побачити план Today.
   Якщо цей ланцюг падає — СТОП: це блокер №1, все інше не має значення.
3. **З телефона.** Весь сценарій однією рукою за реальним посиланням,
   не через DevTools-емуляцію.
4. **Empty state.** Новий користувач бачить пояснення і запрошення до дії, а не порожнечу.
5. **AI-відповідь валідується.** Кривий JSON, зайвий текст навколо, таймаут API —
   ніщо з цього не кладе UI білим екраном.
6. **Секрети.** ANTHROPIC_API_KEY тільки в env-змінній на Vercel.
   `git log -p | grep -i "sk-ant"` — порожньо. У клієнтському бандлі ключа немає.
7. **API-кредити живі.** Баланс не нульовий, кредити заклеймлені й не протухли.
8. **Останній коміт = робоча версія.** Жодних «маленьких фіч» після останнього прогону.
   Робоча версія закомічена? Тоді й тільки тоді можна експериментувати.
9. **Loom.** До 7 хвилин, доступ «anyone with the link», показує проблему → демо → рішення.
10. **Форма здачі.** Обидва посилання (MVP + Loom) прикріплені, відправлено з буфером
    до дедлайну — не о 17:59.

## Формат відповіді
Таблиця: пункт → PASS / FAIL / RISK → що зробити.
Після таблиці — ранжований список фіксів, блокери перші.
Якщо є хоч один FAIL у пунктах 1–2, прямо скажи: «не здавай, спочатку фікс».
