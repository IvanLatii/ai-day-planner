## Що будуємо зараз (Фаза 2 — ядро наскрізь)

Capture → AI-парсинг → Inbox → Today. Один наскрізний сценарій, нічого понад це.
Критерій виходу: наскрізний прогін хаос-тексту з телефона на прод-URL.

## Фінальна схема даних (контракт UI↔AI)

```ts
type Task = {
  id: string
  title: string
  priority: "low" | "medium" | "high"
  due_date?: string          // ISO date, без часу
  time_estimate_min?: number
  tags: string[]             // AI парсить, UI показує чипами; фільтрів немає
  status: "inbox" | "today" | "done"
  source_text: string        // оригінальний фрагмент вводу — показується в картці
  unparsed?: boolean         // true = AI не впевнений; UI показує бейдж "перевір руками"
  created_at: string
}
```

AI повертає СТРОГО JSON-масив таких об'єктів, без жодного тексту навколо.
Відповідь завжди валідується: кривий JSON, зайвий текст, таймаут → fallback, ніколи білий екран.
Fallback: нерозібраний фрагмент → одна задача, `unparsed: true`, `priority: "medium"`,
`source_text` = сам фрагмент.
Сортування в Today — похідне (priority ↓, потім time_estimate_min), окремого поля порядку немає.

## AI-провайдер — ОНОВЛЕНО 19.07 (заміна Anthropic на Gemini)

Оплата Anthropic Console не проходить (українські картки блокуються на боці банку /
регіональне обмеження). Рішення (продуктове, не тимчасовий хак): бренд моделі для
журі не критичний, критична — надійність наскрізного сценарію. Тому:

**Основний провайдер — Gemini API**, модель `gemini-3.5-flash` (безкоштовний тір,
станом на липень 2026; підтверджено пошуком — free tier покриває function calling,
JSON mode, 1M context). Ключ береться на personal Google акаунті через
aistudio.google.com/apikey — без картки, без затримки.

- Переваги для цього сценарію: нативний JSON mode (добре лягає на вимогу "строго
  JSON-масив без зайвого тексту"), офіційний SDK, немає спільного rate-limit
  проблеми безкоштовних моделей на OpenRouter (важливо саме в момент живого демо
  перед журі).
- Ключ ТІЛЬКИ в env-змінній на сервері (перейменувати з `ANTHROPIC_API_KEY` на щось
  на кшталт `GEMINI_API_KEY` або тримати нейтральну назву `AI_API_KEY`), виклик
  через API route. Ніколи в клієнтський код, ніколи в git.
- Архітектура має абстрагувати виклик AI за одним інтерфейсом (умовно
  `parseTasksWithAI(text): Promise<Task[]>`), щоб провайдера можна було підмінити
  (Gemini ↔ Anthropic ↔ мок) без переписування UI/Inbox/Today.

**Мок-режим лишається як fallback**, якщо і Gemini раптом дасть збій під час
підготовки: одна функція повертає мок-масив Task[] за тією ж схемою (з прикладами
unparsed: true, різних priority, tags) — щоб UI перевірявся на всіх станах, не
тільки на "щасливому шляху".

**Anthropic API — не викидаємо остаточно**, лишається можливим шляхом, якщо оплата
раптом пройде (ключ `ai-planner-key` уже створено в Console) — тоді просто інша
реалізація того самого інтерфейсу `parseTasksWithAI`.

## Продуктові принципи (не порушувати під тиском дедлайну)

- Правка в Inbox = 1 тап (чип пріоритету, inline accordion), НЕ форма редагування.
  Це рамковий принцип, а не деталь UI "на потім" — критично для того, щоб продукт
  не відтворював той самий "податок на ведення", від якого він рятує.
- Empty state нового користувача обов'язковий — перше, що побачить журі.
  Порожній екран без нього виглядає як баг.
- Mobile-first: тап-таргети ≥44px, весь сценарій — однією рукою з телефона.
- Спочатку "працює", потім "красиво", але фінальний UI має бути на рівні.
- Один наскрізний сценарій важливіший за будь-яку нову фічу. Не додавати нічого
  поза Capture → Inbox → Today, навіть якщо здається простим.

## Архітектурна вимога — розділити дані й навігацію

Логіка задач (CRUD, виклик AI-парсингу, сортування, localStorage) має жити в
одному окремому місці — модуль чи хук (наприклад `lib/tasks.ts` або `useTasks()`),
незалежно від того, які екрани/роути це показують. Сторінки (`/capture`, `/inbox`,
`/today`) лише споживають цей шар, не містять власної бізнес-логіки.

Навіщо: навігацію (таб-бар, кількість екранів, де саме живе Capture) ми, ймовірно,
захочемо поекспериментувати в Фазі 3 після першого погляду на прод. Якщо дані
відв'язані від екранів — це переставити JSX за пару годин. Якщо змішано —
це переробка. Коштує нуль часу зараз, економить у Фазі 3.

## Потік і правила поведінки (погоджено 19.07)

- Capture приймає БУДЬ-ЯКИЙ горизонт (сьогодні/тиждень/далі) — не просити юзера
  фільтрувати "тільки на сьогодні". Задача без дати = нормальний стан, не помилка.
- Після "Розібрати": лоадер → перехід в Inbox з картками. Результат парсингу НЕ
  показується на екрані Capture — Inbox і є екраном результату.
- "Почати день" переносить в Today все з Inbox, КРІМ задач із майбутньою due_date —
  вони лишаються в Inbox і "дозрівають" до свого дня (вечірнє планування на завтра
  працює само собою). Це один if по due_date.
- Стартовий екран при відкритті — Today (не Capture). Найчастіший намір — "що далі?".
  Новий користувач: Today показує empty state із CTA, що веде на Capture.
- Дії на картці Today без провалювання: done-чекбокс, чип пріоритету, "повернути в Inbox".
  Окремої сторінки задачі у v1 немає.
- Правила дат у промпті: ЗАВЖДИ передавати сьогоднішню дату в запит (інакше "завтра"
  нерозв'язне); неоднозначна дата → найближче майбутнє; справді незрозуміло →
  через відсутність due_date, НЕ вигадувати. Крива дата гірша за відсутню.
- Голос/мікрофон — НЕ зараз (апгрейд: диктування наповнює те саме текстове поле,
  ланцюг далі ідентичний; додається в advanced без зміни архітектури).

## Обробка збою AI — два різні режими (погоджено 19.07)

`api/parse-tasks/route.ts` повертає дискриміновану відповідь:
`{ ok: true, fragments }` | `{ ok: false, reason }`.

- Повний провал виклику (мережа, таймаут, невалідний ключ, порожня відповідь) →
  `ok:false`. UI Capture показує банер "Спробувати ще раз", текст ЛИШАЄТЬСЯ в полі,
  задача НЕ створюється. Керований стан, не білий екран.
- Відповідь прийшла, але крива (побитий JSON / окремий фрагмент не розібрався) →
  `ok:true` з валідними задачами; нерозібране стає `unparsed:true` з бейджем.
  Рятуємо все, що вдалось розпарсити, решту не втрачаємо.

Тобто unparsed:true — це НЕ реакція на "модель мовчить", а на "модель відповіла,
частина незрозуміла". Не плутати два режими.
