# Голосове введення — Блок 2 (Web Speech API, без фолбеку)

## Контекст

Capture-екран (`src/app/capture/page.tsx`) сьогодні приймає тільки текст у textarea →
кнопка «Розібрати на задачі» → `/api/parse-tasks`. ТЗ вимагає додати мікрофон біля
поля «Що в голові?»: тап диктує голосом у те саме поле, ланцюг далі — без змін.

Голос — апгрейд поверх робочого текстового шляху, не фундамент (CLAUDE.md). Тому в
цьому блоці робимо **лише** Web Speech API (`webkitSpeechRecognition`). Фолбек через
MediaRecorder + Gemini-транскрипцію свідомо НЕ будуємо зараз — рішення 20.07: спершу
задеплоїти й протестувати Web Speech на реальному iPhone Safari (середовище розробки
не має справжнього WebKit/Safari, тож попереднє тестування тут ненадійне), і братися
за фолбек лише якщо реальний тест покаже, що він потрібен.

## Архітектура

Вся логіка Web Speech інкапсульована в хук `useVoiceDictation`
(`src/lib/voice/useVoiceDictation.ts`), а не в самій сторінці:

```
useVoiceDictation({ onTranscript: (text: string) => void }): {
  supported: boolean;
  listening: boolean;
  start: () => void;
  stop: () => void;
  error: string | null;
}
```

`capture/page.tsx` викликає `start`/`stop` і рендерить кнопку; хук ховає роботу з
подіями `result`/`error`/`end` браузерного API. Якщо пізніше знадобиться
MediaRecorder-фолбек, він підключиться за тим самим інтерфейсом (`start`/`stop`/
`listening`) — сторінку чіпати не доведеться.

## Feature detection

При монтуванні: `window.SpeechRecognition || window.webkitSpeechRecognition`.

- **Немає підтримки** → `supported: false` → **кнопка мікрофона взагалі не
  рендериться** (не задизейблена — правка користувача 20.07: приховати повністю,
  а не показувати неактивною). Текстовий шлях лишається єдиним і повноцінним.
- **Є підтримка** → кнопка рендериться в звичайному (idle) стані.

## Стан і потік

1. **idle → listening** (тап на мікрофон): створюється `SpeechRecognition` з
   `lang: 'uk-UA'`, `continuous: true`, `interimResults: true`, викликається
   `.start()`. Кнопка мікрофона в тому ж слоті замінюється на індикатор
   «● Слухаю…» (зліва) + кнопку «Стоп» (справа) — без стрибків лейауту.
2. Кожен **фінальний** `result` дописується в textarea через пробіл **після**
   наявного тексту (не замінює написане раніше — рішення 20.07).
3. **listening → idle** (тап «Стоп»): `recognition.stop()`, накопичений текст
   лишається в полі.
4. **error-обробка:**
   - `not-allowed`, `audio-capture`, `network`, несподіваний `aborted` →
     повернення в idle + інлайн-банер (той самий стиль, що вже є для помилок
     `/api/parse-tasks`): «Не вдалося розпізнати голос. Спробуй ще раз або допиши
     текстом».
   - `no-speech` (просто мовчав) → тихе повернення в idle, без банера — це не
     помилка.
   - Несподіваний `end` (браузер сам зупинив розпізнавання після паузи) →
     повернення в idle, щоб кнопка не «зависала» в «Слухаю…».

## UI-розміщення

Кругла кнопка мікрофона в куті textarea (top-right, тап-таргет ≥44px), не окремий
рядок над/під полем. У стані «Слухаю…» той самий кут показує пульсуючу крапку +
текст ліворуч і кнопку-квадратик «Стоп» на місці мікрофона.

## Тестування

Web Speech — браузерний API, юніт-тестами не покривається (немає jsdom-полізіла,
який відтворює реальну поведінку). Перевірка руками в прев'ю-браузері (підтримує
SpeechRecognition): кнопка з'являється лише коли API підтримується, диктування
дописує текст у поле, «Стоп» зупиняє, помилки не кладуть UI. Остаточний
приймальний тест — реальний iPhone Safari, це поза межами того, що можна
перевірити із середовища розробки.

## Явно поза скоупом цього блоку

- MediaRecorder + Gemini-транскрипція (фолбек) — окремий блок, тільки якщо
  реальний тест на iPhone покаже, що Web Speech там глючить.
- Стрімінгова поява карток під час диктування (вирішено не брати, V2).
- Автоматичне визначення мови (мова розпізнавання фіксована `uk-UA`).
